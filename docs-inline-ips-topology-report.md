현재 토폴로지는 `client -> router -> ips(bridge) -> server` 구조로 재설계되었다. `client`는 `client_net(10.10.0.0/24)`에만 연결되고 `gotestwaf` 송신 역할만 맡는다. 새로 추가된 `router` 컨테이너는 `client_net`과 `router_net(10.20.30.0/24)` 양쪽에 붙어 있으며, `client_net` 쪽 주소는 `10.10.0.254`, `router_net` 쪽 Docker 주소는 `10.20.30.254`이다. 이 `router`는 실행 시 upstream 인터페이스에 서비스용 주소 `10.20.0.1/24`를 추가하고, `client`가 `10.20.0.0/24` 대역으로 가는 트래픽을 받아 `ips` 앞으로 보낸다. 이 구조에서 L3 게이트웨이 역할은 `router`가 전담하고, `ips`는 더 이상 라우터로 동작하지 않는다. Docker의 user-defined bridge 네트워크는 동일한 `10.20.0.0/24`를 두 번 겹쳐 둘 수 없으므로, transit 링크는 같은 `10.20.x` 계열이지만 별도 서브넷 `10.20.30.0/24`로 분리했다.

`ips`는 이제 `router_net`과 `server_net(10.20.0.0/24)`에 붙은 순수 inline 장비다. `iptables_tproxy.sh`는 두 개의 data-plane 인터페이스를 자동 탐지해서 `br0` 브리지를 만들고, 해당 포트의 기존 주소를 비운 뒤 `BRIDGE_CIDR=10.20.0.20/24`를 브리지에 부여한다. 동시에 `br_netfilter`, `xt_TPROXY`, `nf_tproxy_ipv4`, `xt_socket` 준비 상태와 `net.bridge.bridge-nf-call-iptables=1`, `net.ipv4.conf.all.route_localnet=1`, `net.ipv4.conf.all.rp_filter=0` 값을 검사한다. 그 다음 `CLIENT_IP=10.10.0.10`, `SERVER_IP=10.20.0.10`, `SERVER_PORT=80`에 맞는 PREROUTING TPROXY 규칙을 설치하고, fwmark `0x1`이 local table `100`으로 향하도록 설정한다. 이 구성이 맞아야 `client`에서 `10.20.0.10:80`으로 보낸 요청이 `ips` 유저스페이스 리스너 `50080`으로 전달된다.

유저스페이스 리스너는 [`ips/src/inline/main.c`](/home/doogunwo/training/Mini-IPS/ips/src/inline/main.c#L131)에 구현되어 있다. 이 코드는 `accept()`한 소켓에 대해 [`getsockname()`](/home/doogunwo/training/Mini-IPS/ips/src/inline/main.c#L173)으로 원래 목적지 주소를 얻고, 그 값을 그대로 [`connect_upstream(&local_addr)`](/home/doogunwo/training/Mini-IPS/ips/src/inline/main.c#L186)로 넘긴다. 따라서 TPROXY로 유입된 연결은 원래 향하던 서버 주소를 유지한 채 업스트림으로 연결된다. 이후 `poll()` 기반 relay가 클라이언트와 서버 사이의 양방향 TCP 바이트 스트림을 중계한다. 즉 네트워크 구조가 inline bridge로 바뀌더라도 유저스페이스 로직은 "transparent accept -> original destination recovery -> upstream relay" 흐름을 그대로 유지한다.

실행 순서는 명확하다. 먼저 `sudo docker compose up -d server`, `sudo docker compose --profile manual up -d --build router ips`, `sudo docker compose --profile manual build client`로 컨테이너를 준비한다. 그 다음 `ips` 컨테이너 안에서 `./iptables_tproxy.sh .env up`으로 bridge + TPROXY 정책을 적용하고, `./build/bin/inline-ips`로 리스너를 띄운다. 마지막으로 호스트에서 `sudo docker compose --profile manual run --rm client --url=http://10.20.0.10:80 ...`를 실행한다. 정상 동작의 기준은 `inline-ips`가 `accepted client=10.10.0.10 ... original_dst=10.20.0.10:80`를 출력하고, 동시에 `iptables -t mangle -L PREROUTING -n -v`의 TPROXY 카운터가 증가하는 것이다. 이 두 조건이 만족되면 `client -> router -> ips bridge -> server` 데이터 경로와 TPROXY 유저스페이스 처리 경로가 함께 성립한 것이다.
